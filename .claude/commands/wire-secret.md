## Context
- Secrets are managed via SOPS and stored in `sops/` directory within this repo
- Decrypted secrets are placed at `~/.config/sops/secrets/` (or `~/.config/sops/secrets/<hostname>/` for host-specific secrets)
- SOPS secret files are YAML format (e.g., `gitlab.yaml`, `aws.yaml`, `bedrock.yaml`)
- The `sops-nix` home-manager module handles decryption during activation

## Your Task

Wire a SOPS-managed secret into a `.nix` configuration file.

### Steps

1. Locate the SOPS secret file in the `sops/` directory and read its structure (keys, format)
2. Determine the decrypted secret path:
   - Host-specific: `~/.config/sops/secrets/<hostname>/<secret>.yaml`
   - Shared: `~/.config/sops/secrets/<secret>.yaml`
3. Read the target `.nix` file that needs to reference the secret
4. Wire the secret by:
   - Adding SOPS secret declaration in the home-manager sops config
   - Referencing the decrypted secret path in the target application config
   - Using `builtins.readFile` or template interpolation as appropriate
5. Verify the secret path matches the actual file location in `sops/`

### Guardrails
- Never hardcode secret values in `.nix` files — always reference SOPS paths
- Check whether the secret file has moved (common pattern: from shared to host-specific path)
- After wiring, remind the user to rebuild and verify the secret appears in the target config file
- If a generated config file (like `~/.config/glab-cli/config.yml`) doesn't contain the expected token after build, check that the SOPS activation step runs BEFORE the config generation step
- Host-specific secrets should use the hostname in the path (dendritic pattern)
